# 12 编码BOOLEAN类型
**12.1** BOOLEAN类型的值 ***必须*** 编码为由单个比特组成的位域。

**12.2** 该比特 ***必须*** 设置为1表示TRUE，设置为0表示FALSE。 

**12.3** 该位域 ***必须*** 附加到字段列表中，且不包含长度决定因子。
# 13 编码INTEGER类型

注1 ——（ALIGNED变体教程）允许将所有值编码为一个八位组或更少的取值范围应放入一个最小尺寸的比特字段中且不包含长度计数。允许将所有值编码为两个八位组的范围，应放入两个八位对齐的比特字段中，且不包含长度计数。否则，该值编码为最少数量的八位组（使用非负二进制整数或2的二进制补码整数编码），并添加一个长度决定因子。在此情况下，如果整数值可以编码为少于127个八位组（作为来自可能确定的任何下限的偏移量），并且没有有限的上限和下限，则长度决定因子为一个八位组；否则，长度编码为所需的最少比特数。其他情况在实际应用中并无意义，但为了完整性而予以规定。

注2 —— （UNALIGNED变体教程）被约束整数以表示该范围所需的最少比特数进行编码，而不考虑其大小。无约束整数则按注1所述进行编码。

**13.1** 如果在integer类型的约束规范中存在扩展标记，则 ***必须*** 向字段列表中添加长度为1比特的位域。如果待编码的值不在扩展根（注释，...之前）的范围内，则该位 ***必须*** 设置为1，否则设置为0。在前一种情况下，该值 ***必须*** 作为 *无约束整数值* 添加到字段列表中，如13.2.4至13.2.6中所规定，从而完成此过程。在后一种情况下，该值 ***必须*** 按照不存在扩展标记的方式进行编码。

**13.2** 如果integer类型的约束规范中不存在扩展标记，则适用以下规则。

**13.2.1** 如果 PER可见约束将整数值限定为单一固定值，则 ***禁止*** 向字段列表中添加任何内容，本流程结束。

**13.2.2** 如果PER可见约束将整数值限定为约束整数，则 ***必须*** 按照 11.5 节（约束整数的编码）的过程将其转换为一个字段，随后应用 13.2.5 至 13.2.6 节的过程。

**13.2.3** 如果PER可见约束将整数值限定为半约束整数，则 ***必须*** 按照 11.7 节（半约束整数的编码）的过程将其转换为一个字段，随后应用 13.2.6 节的过程。

**13.2.4** 如果PER可见约束既未将该整数限定为约束整数，也未限定为半约束整数，则 ***必须*** 按照 11.8 节（无约束整数的编码）的过程将其转换为一个字段，随后应用 13.2.6 节的过程。

**13.2.5** 如果用于将整数值编码为字段的过程未产生不定长情形（参见 11.5.7.4 和 11.8.2），则 ***必须*** 将该字段追加到字段列表中，本流程结束。

**13.2.6** 否则（即为不定长情形），***必须*** 调用 11.9 节的过程，将该字段追加到字段列表，并在其前面添加以下二者之一：
- a) 一个约束长度决定因子 len（由 11.5.7.4 确定），条件是：PER 可见约束对该类型限定了有限的上界与下界，且若该类型可扩展，则值位于扩展根的取值范围之内。该长度决定因子所使用的下界 lb  ***必须*** 为 1，上界 ub ***必须*** 为容纳该整数取值范围所需的八位位组数量。
	>注：取值 `foo INTEGER (256..1234567) ::= 256` 在ALIGNED变体中的编码结果形如 `00xxxxxx00000000`，其中每个 x 表示一个可能存在也可能不存在的零填充比特，具体取决于长度字段在八位位组中的起始位置。例如，若长度从八位位组边界开始，则编码为 `00 xxxxxx 00000000`；若从某八位位组的最低两位（比特 2 和比特 1）开始，则编码为 `00 00000000`。

- b) 一个取值等于 len 的无约束长度决定因子（由 11.7 和 11.8 确定），条件是：PER 可见约束未对该类型限定有限的上界与下界，或者该类型可扩展且值不在扩展根的取值范围之内。

# 14 编码ENUMERATED类型
>注：（教程）不带扩展标记的枚举类型，其编码方式等同于子类型约束中不含扩展标记的约束整数。这意味着在实际应用中，枚举类型几乎总是以能够表示所有枚举项所需的最少比特数构成的位域进行编码。若存在扩展标记，且该值不属于扩展根，则按照 常规较小的非负整数 进行编码。

**14.1** 枚举根中的枚举项 ***必须*** 按其枚举值升序排列，并为其分配枚举索引：第一个枚举项索引为0，第二个为1，依此类推直至最后一项。扩展追加项（始终按升序定义）也 ***必须*** 分配枚举索引：第一个追加枚举项索引为0，第二个为1，依此类推直至最后一项。

>注：《REC ITU-T X.680 | ISO/IEC 8824-1》 要求，每一个后续的扩展追加项的枚举值均 ***必须*** 大于前一项。

**14.2** 若枚举类型定义中不存在扩展标记，则 ***必须*** 对枚举索引进行编码。其编码方式 ***必须*** 等同于一个不存在扩展标记的约束整数类型的值，该类型下限为0，上限为该类型关联的最大枚举索引，本过程至此结束。

**14.3**  若存在扩展标记，则 ***必须*** 向字段表中添加一个长度为1比特的位域。若待编码值不属于扩展根，则该比特 ***必须*** 置1，否则置0。在前一种情况下，扩展追加项 ***必须*** 按照14.1排序，并将该值作为 常规小非负整数 添加至字段表，其值为该追加枚举项的枚举索引，且下限lb设为0，本过程至此结束。在后一种情况下，***必须*** 按照14.2的规定，视同不存在扩展标记对该值进行编码。

>注：不存在可应用于枚举类型且对本编码规则可见的PER可见约束。

# 16 编码BIT STRING类型
>注——（教程）被约束为固定长度且不大于16比特的BitString不会触发八位位组对齐。更长的比特串在 ALIGNED 变体中按八位位组对齐。若长度由约束固定且上限小于64K，则不会出现显式长度编码；否则包含长度编码，该编码可采用前述长度编码的任意格式，包括对长比特串进行分片编码。

**16.1** PER 可见约束仅用于约束比特串的长度。

**16.2** 若不存在 PER 可见约束，且适用 《Rec. ITU-T X.680 | ISO/IEC 8824-1》 第22.7款，则该值 ***必须*** 在不携带尾随0比特的情况下编码（注：这意味着不含1比特的值始终编码为空比特串）。

**16.3** 若存在 PER 可见约束，且适用 《Rec. ITU-T X.680 | ISO/IEC 8824-1》 第22.7款（即比特串类型通过“NamedBitList”定义），则**必须**根据需要添加或删除尾随0比特后再对该值编码，确保传输值的长度为可承载该值且满足有效长度约束的最小长度。

**16.4** 设比特串中的最大比特数（由长度相关的 PER 可见约束确定）为`ub`，最小比特数为`lb`。若无有限最大值，则称`ub`未设定。若无最小长度约束，则`lb`取值为0。设待编码的实际比特串值长度为`n`比特。

**16.5** 当按照16.6至16.11款的规定将比特串值放入比特域时，比特串值的首比特 ***必须*** 置于比特域的首位，比特串值的最低位比特 ***必须*** 置于比特域的最低位。

**16.6** 若该类型针对 PER 编码是可扩展（见10.3.9），则 ***必须*** 向字段列表添加一个由单比特构成的比特域。若本编码的长度超出扩展根范围，该比特 ***必须*** 置1，否则置0。在前一种情况下，***必须*** 调用16.11款，将长度作为半约束整数加入字段列表，随后跟随比特串值。在后一种情况下，长度与值 ***必须*** 按照约束中不存在扩展的方式编码。

**16.7** 若比特串类型的约束说明中不存在扩展标记，则适用16.8至16.11款。

**16.8** 若比特串被约束为零长度（`ub`等于0），则**禁止**对其编码（不向字段列表添加任何内容），本条款过程完成。

**16.9** 若比特串的所有取值均被约束为同一长度（`ub`等于`lb`），且该长度不大于16比特，则**必须**将该比特串放入长度为约束值`ub`的比特域中，该比特域 ***必须*** 追加至字段列表且不携带长度决定因子，本条款过程完成。

**16.10** 若比特串的所有取值均被约束为同一长度（`ub`等于`lb`），且该长度大于16比特但小于64K比特，则 ***必须*** 将该比特串放入长度为`ub`（不必为8比特的整数倍）的比特域中（ALIGNED 变体下按八位位组对齐），该比特域 **必须** 追加至字段列表且不携带长度决定因子，本条款过程完成。

**16.11** 若16.8至16.10款均不适用，则 ***必须*** 将该比特串放入长度为`n`比特的比特域中（ALIGNED 变体下按八位位组对齐），并 ***必须*** 调用11.9款流程，将该`n`比特的比特域（ALIGNED 变体下按八位位组对齐）加入字段列表；在其之前 ***必须*** 添加长度为`n`比特的长度指示符：若`ub`已设定且小于64K，则作为约束整数编码；若`ub`未设定，则作为半约束整数编码。`lb`取值按前述规则确定。

>注——对于无约束或`ub`较大的场景，在长度达到16K、32K、48K或64K比特后启用分片编码。
# 17 编码OCTET STRING类型
>注——长度小于或等于两个八位位组的定长八位位组串，不按八位位组对齐。在 ALIGNED 变体中，其余所有八位位组串均按八位位组对齐。若定长八位位组串长度小于 64K，则编码时不包含长度八位位组。对于无约束八位位组串，长度将被显式编码（必要时进行分片）。

**17.1** PER 可见约束仅可约束八位位组串的长度。

**17.2** 设八位位组串中的最大八位位组数量（由关于长度的 PER 可见约束确定）为“ub”，最小八位位组数量为“lb”。若不存在有限最大值，则称“ub”未设置。若对最小值无约束，则“lb”取值为 0。设待编码的实际八位位组串值的长度为 n 个八位位组。

**17.3** 若该类型针对 PER 编码是可扩展的（见 10.3.9），则 ***必须*** 向字段列表中添加一个由单个比特构成的比特域。若本编码的长度不在扩展根的取值范围内，该比特 ***必须*** 置为 1，否则置为 0。在前一种情况下，***必须*** 调用 17.8 条款，将长度作为半约束整数加入字段列表，随后附加八位位组串的值。在后一种情况下，长度与值 ***必须*** 按照约束中不存在扩展标记的方式进行编码。

**17.4** 若八位位组串类型的约束规范中不存在扩展标记，则适用 17.5 至 17.8 条款。

**17.5** 若八位位组串被约束为零长度（ub 等于 0），则 ***禁止*** 对其进行编码（不向字段列表添加任何内容），本条款过程完成。

**17.6** 若八位位组串的所有取值均被约束为同一长度（ub 等于 lb），且该长度小于或等于两个八位位组，则该八位位组串 ***必须*** 放入一个比特数等于约束长度 ub 乘以 8 的比特域中，该比特域 ***必须*** 追加至字段列表，不携带长度决定因子，本条款过程完成。

**17.7** 若八位位组串的所有取值均被约束为同一长度（ub 等于 lb），且该长度大于两个八位位组但小于 64K，则该八位位组串 ***必须*** 放入一个长度为约束长度 ub 个八位位组的比特域中（ALIGNED 变体下按八位位组对齐），该比特域 ***必须*** 追加至字段列表，不携带长度决定因子，本条款过程完成。

**17.8** 若 17.5 至 17.7 均不适用，则该八位位组串 ***必须*** 放入长度为 n 个八位位组的比特域中（ALIGNED 变体下按八位位组对齐），并 ***必须*** 调用 11.9 条款流程，将该 n 个八位位组的比特域（ALIGNED 变体下按八位位组对齐）加入字段列表；在其之前 ***必须*** 添加一个等于 n 个八位位组的长度决定因子：若 ub 已设定，则作为约束整数编码；若 ub 未设定，则作为半约束整数编码。lb 取值按前述规则确定。

>注——分片过程可在达到 16K、32K、48K 或 64K 个八位位组后启用。

# 18 编码NULL类型
>注：（教程）NULL类型本质上是一个占位符，仅在CHOICE类型或可选的SET/SEQUENCE组件场景下具有实际含义。在CHOICE中对NULL的标识或其作为可选元素出现的判定，在本编码规则中无需使用表示NULL类型的八位位组即可执行。因此，NULL值不会为编码增加任何八位位组。

对于NULL值，禁止向字段表添加任何内容。
# 19 编码SEQUENCE类型
>注：（教程）序列类型以一段位图形式的前导开头。如果序列类型没有扩展标记，那么该位图仅记录类型中默认组件和可选组件是否存在，编码为固定长度的比特域。如果序列类型带有扩展标记，那么该位图前会先有一个单比特，用于指明编码中是否实际出现了扩展增补项的值。只要前导长度小于 64K 比特，就不带长度决定因子进行编码；否则编码长度决定因子以实现分片。前导之后依次是各个组件的编码字段。如果存在扩展增补项，那么在编码第一个扩展增补项之前，***必须*** 先编码次类型中将编码的扩展增补项数量（编码为常规较小长度），随后紧跟一个长度与该数量相等的位图，记录每个扩展增补项是否出现。之后再对扩展增补项进行编码，每个扩展增补项按照开放类型字段的值来编码。

**19.1** 如果SEQUENCE类型在“ComponentTypeLists”或“SequenceType”产生式中包含扩展标记，***必须*** 先向字段列表添加一个长度为 1 的比特域。如果本次编码中存在扩展增补项的值，该比特 ***必须*** 置 1，否则置 0（下文将该比特称为“扩展比特”）。如果“ComponentTypeLists”或“SequenceType”产生式中没有扩展标记，则 ***禁止*** 添加扩展比特。

**19.2** 如果SEQUENCE类型的扩展根中有 `n` 个被标记为 OPTIONAL 或 DEFAULT 的组件，***必须***生成一个 `n` 比特的比特域并加入字段列表。该比特域的各位按顺序编码SEQUENCE类型中每个可选或默认组件编码是否出现。比特值为 1 表示该组件编码***必须*** 出现，比特值为 0 表示该分量编码 ***必须*** 不存在。前导的首比特 **必须** 编码第一个可选或默认组件是否出现，末尾比特 ***必须*** 编码最后一个可选或默认组件是否存在。

**19.3** 如果 `n` 小于 64K，该比特域 ***必须*** 被追加到字段列表。如果 `n` 大于或等于 64K，***必须*** 调用 11.9 的过程将该 `n` 比特的比特域加入字段列表，并在其前面附加长度决定因子（值为 `n` 比特），其编码为约束整数，其中 `ub` 和 `lb` 均设为 `n`。

>注：在这种情况下，长度处理过程将忽略`ub` 和 `lb`。此处调用该过程仅为对大型前导码进行分片。该情况预计极少出现。

**19.4** 前导码之后 ***必须*** 依次跟上SEQUENCE值中存在的各个组件的字段列表。

**19.5** 对于 CANONICAL-PER，如果待编码值为默认值，被标记为 DEFAULT 的组件的编码 **必须** 始终不出现。对于 BASIC-PER，如果待编码值为简单类型的默认值（见 3.7.25），被标记为 DEFAULT 的组件的编码 ***必须*** 始终不出现；其他情况下，是否对其编码由发送方自行选择。

**19.6** 如果没有扩展比特或扩展比特为 0，编码到此完成。如果扩展比特存在且被置为 1，***必须*** 应用后续流程。

**19.7** 设将被编码的类型中的扩展增补项数量为 `n`，***必须*** 生成一个 `n` 比特的比特域并加入字段列表。该比特域的各位按顺序编码将被编码类型中每个扩展增补项的编码是否出现。比特值为 1表示该扩展增补项编码**必须**存在，比特值为 0表示**必须**不存在。该比特域的首比特 ***必须*** 编码第一个扩展增补项是否存在，末尾比特 ***必须8** 编码最后一个扩展增补项是否存在。

>注：如果声明符合某一版本规范，那么 `n` 的值 ***必须*** 始终等于该版本中的扩展增补项数量。

**19.8** ***必须*** 调用 11.9 的流程将该 `n` 比特的比特域加入字段列表，并在其前面附加长度决定因子（值为 `n`），其编码为常规较小长度。

>注：`n` 不可能为 0，因为只有在至少有一个扩展增补项需要编码时才会执行本过程。

**19.9** 其后**必须**依次跟上包含各个出现的扩展增补项的编码的字段列表。每个属于“ComponentType”的扩展增补项（即不属于“ExtensionAdditionGroup”）***必须*** 按照 11.2.1 的规定，如同开放类型（OPEN）字段的值一样进行编码。每个属于“ExtensionAdditionGroup”的扩展增补项 ***必须*** 先按照 19.2 至 19.6 规定的 SEQUENNCE 类型进行编码，再按照 11.2.1 的规定如同开放类型字段的值一样进行编码。如果“ExtensionAdditionGroup”的所有组件值均缺失，该“ExtensionAdditionGroup” ***必须*** 按缺失扩展增补项编码（即 19.7 所述位图中的对应比特 ***必须*** 置 0）。

>注 1——如果“ExtensionAdditionGroup”包含被标记为 OPTIONAL 或 DEFAULT 的组件，该“ExtensionAdditionGroup” ***必须*** 以一个位图开头，指明每个被被标记为 OPTIONAL 或 DEFAULT组件的值是否存在。

>注 2——在扩展标记对之后定义的“RootComponentTypeList”中的组件，其编码方式等同于将其定义在扩展标记对紧前的位置。

# 20 编码SEQUENCE-OF类型
**20.1** PER可见约束能够对序列类型的组件数量进行约束。

**20.2** 设序列中由 PER可见约束确定的组件最大数量为 ub，最小数量为 lb。若不存在有限的最大值，或 ub 大于或等于 64K，则称 ub 为未设置。若对最小值无约束，则 lb 取值为 0。设待编码的实际序列值中的组件数量为 n 个。

**20.3** 序列中每个组件的编码将生成若干字段，并追加到该序列类型的字段列表中。

**20.4** 若存在PER可见约束且其中包含扩展标记，则 ***必须*** 在字段列表中添加一个长度为 1 的比特位。若本次编码中的组件数量不在扩展根的范围内，该比特 ***必须*** 置为 1，否则置为 0。在前一种情况下，***必须*** 调用 11.9，将长度决定因子作为半约束整数添加到字段列表，随后跟随各组件的值。在后一种情况下，长度与值 ***必须*** 视为不存在扩展标记来进行编码。

**20.5** 若组件数量固定（ub 等于 lb）且 **ub** 小于 64K，则该序列 ***禁止*** 设置长度决定因子，各组件的字段 ***必须*** 依次追加到该序列的字段列表中。

**20.6** 否则，***必须*** 调用 11.9 的过程，将由n个组件生成的字段列表添加到其字段列表，并在其前面添加长度决定因子：若 ub 已设置，则以约束整数形式表示数量为 n个组件；若 ub 未设置，则以半约束整数形式表示。lb 采用上述确定的值。

>注1 – 分片过程可能在 16K、32K、48K 或 64K 个组件之后生效。

>注2 – 分片的分界点位于字段之间。分界点之前的比特数不一定是 8 的整数倍。

# 21 编码SET类型

SET类型 ***必须*** 将其“RootComponentTypeList” 中的元素按照 《Rec. ITU‑T X.680 | ISO/IEC 8824‑1》 第8.6款规定的规范顺序排序；此外，为确定当一个或多个组件为无标签的CHOICE类型时各组件的编码顺序，每个无标签CHOICE类型的排序方式视同其拥有一个标签，该标签等于该CHOICE类型的“RootAlternativeTypeList”中的或其内部嵌套的任意无标签CHOICE类型中的最小标签。出现在“RootComponentTypeList”中的集合元素 ***必须*** 按其被声明为SEQUENCE类型的方式进行编码。出现在“ExtensionAdditionList”中的集合元素 ***必须*** 按照19.9款规定的SEQUENCE类型组件方式进行编码（即按照其被定义的顺序编码）。

示例——在下例中，假定标签环境为隐式标签（IMPLICIT TAGS）：

```
		A ::= SET
		{
			a	[3] INTEGER,
			b	[1] CHOICE
			{
				c	[2] INTEGER,
				d	[4] INTEGER
			},
			e	CHOICE
			{
				f	CHOICE
				{
					g	[5] INTEGER,
					h	[6] INTEGER
				},
				i		CHOICE
				{
					j	[0] INTEGER
				}
			}
		}
```

该集合中各组件的编码顺序始终为 e、b、a，这是因为标签 \[0] 排序最靠前，其后是\[1]，再之后是 \[3]。

# 22 编码SET-OF类型
**22.1**  对于 CANONICAL-PER，SET-OF 类型各组件值的编码 ***必须*** 按升序排列。各组件编码需按比特串进行比较：在其尾部填充至多7个0比特以对齐八位位组边界，必要时向较短的比特串添加全0八位位组，使其长度与较长者一致。

>注——为排序而添加的任何填充比特或填充八位位组 ***禁止*** 出现在实际编码中。

**22.2**  对于 BASIC-PER，SET-OF  ***必须*** 按照其被声明为 SEQUENCE-OF 类型的方式进行编码。

# 23 编码CHOICE类型
>注——（教程）CHOICE类型通过编码一个指定的所选备选项的索引来进行编码。该索引按照约束整数的方式编码（除非CHOICE类型中存在扩展标记，此时索引按常规小非负整数编码），因此通常占用一个固定长度的位域，其比特数为编码该索引所需的最小比特数。（尽管理论上该索引可以任意大。）索引之后是所选备选项的编码，属于扩展增补项的备选项按照其为OPEN类型字段的值的方式进行编码。若CHOICE类型仅有一个备选项，则索引部分无编码。

**23.1**  CHOICE类型的编码不受 PER 可见约束的影响。

**23.2**  CHOICE类型的每个组件均关联一个索引，按照 《Rec. ITU‑T X.680  | ISO/IEC 8824‑1》 第8.6款规定的规范顺序， CHOICE类型扩展根中的第一个备选项索引值为0，第二个为1，依此类推直至CHOICE类型扩展根中的最后一个组件。"ExtensionAdditionAlternativesList" 中的每个"NamedType" 类似地分配索引值，与扩展根的分量一致从0开始。设`n`为扩展根中最大的索引值。

>注——《Rec. ITU‑T X.680  | ISO/IEC 8824‑1》 第29.7款要求，每个后续的扩展增补项的标签值 ***必须*** 大于"ExtensionAdditionAlternativesList"中上一次添加项的标签值。

**23.3** 为了对包含无标签CHOICE类型的CHOICE备选项的规范排序，每个无标签CHOICE类型 ***必须*** 按照如下方式排序：视同其拥有一个标签，该标签等于该CHOICE类型或其内部嵌套的任意无标签CHOICE类型的扩展根中的最小标签。

**23.4**  若CHOICE类型的扩展根中仅有一个备选项，且选中该备选项，则索引部分 ***禁止*** 编码。

**23.5**  若CHOICE类型在 "AlternativeTypeLists" 产生式中包含扩展标记，则必须先向字段列表添加一个长度为1的位域。若编码中存在扩展增补项的值，该比特必须置1，否则置0。（下文将该比特称为“扩展比特”。）若 "AlternativeTypeLists"产生式中无扩展标记，则 ***禁止*** 添加扩展比特。

**23.6**  若不存在扩展比特，则所选备选项的选择索引 ***必须*** 按照第13条的过程编码至一个字段中，视同其为约束在0至`n`范围的整数类型的值（其子类型约束中无扩展标记），且该字段 ***必须*** 追加至字段列表。其后 ***必须*** 跟随所选备选项的字段，本条款过程至此完成。

**23.7**  若存在扩展比特且所选备选项位于扩展根内，则所选备选选项的选择索引 ***必须*** 按照第13条的过程编码（视为不存在扩展标记）。其后 ***必须*** 跟随所选备选项的字段，本条款过程至此完成。

23.8  若存在扩展比特且所选备选项不在扩展根内，则所选备选项的选择索引 ***必须*** 编码为常规小非负整数，其中`lb`设为0，且该字段 ***必须*** 追加至字段列表。其后 ***必须*** 跟随包含所选备选项编码的字段列表，该备选项按照11.2款规定视同开放类型字段的值进行编码，本条款过程至此完成。

>注——CHOICE类型扩展增补项定义中的版本括号对"ExtensionAdditionAlternatives"的编码方式无影响。
